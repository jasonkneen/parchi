Parchi — What’s Going On, Why It Exists, How It Works, and How to Structure It Next

Repo: browser-ai Core shape: Multi-runtime browser agent Primary bottleneck: orchestration concentration Immediate direction: split by runtime responsibilities

1) Systems We Have (WHAT + WHY + HOW)

Each card answers what it is, why it exists, and how it currently behaves.

Extension Background Runtime

What: Orchestrator and tool runtime for agent turns.

Why: Keep model/tool execution close to browser APIs.

How: BackgroundService handles run lifecycle, model routing/fallbacks, tool execution, relay events.

Sidepanel UI Runtime

What: User-facing chat, settings, account, history, usage.

Why: Unified control plane for interactive agent work.

How: Prototype-augmented SidePanelUI modules consume runtime messages and render timeline/state.

Backend (Convex + Stripe)

What: Auth, paid routing, subscriptions, checkout/provisioning.

Why: Enable managed access (credits/subscription) and secure server-side operations.

How: Convex HTTP routes + actions; payment/provisioning paths in payments.ts.

Relay Service + Daemon

What: Local JSON-RPC bridge between extension and external clients.

Why: Makes browser agent callable from CLI/native host/electron workflows.

How: WS endpoint with authenticated token, run event buffering, run wait semantics.

CLI / Electron / Shared Contracts

What: Operator tooling and shared schemas/types.

Why: Keep protocol compatibility and reduce duplicated assumptions.

How: CLI commands map to relay RPC; @parchi/shared defines runtime + JSON-RPC + plan/tool contracts.

3) Runtime Path (HOW IT RUNS)

Critical path for one user turn and where retries/guardrails live.
Step 1

User Turn Starts

Sidepanel sends user_message with session history, selected tabs, and optional recorded context.

Step 2

Run Orchestration

Background allocates run/session state, emits status, normalizes conversation, attaches guardrails.

Step 3

Model Route Selection

Route chosen via BYOK/OAuth/paid proxy, then model call with streaming + fallback/retry controls.

Step 4

Tool Execution Loop

Tool calls run in browser APIs; permissions checked, tab/session scoped, results emitted to UI.

Step 5

Finalization

Assistant final response + usage + benchmark metadata sent; context compaction may run after.

Step 6

External Access

Relay mirrors events (run.event/run.done) so CLI/native/electron can monitor and control runs.

4) Fundamentals to Keep (WHY THIS WORKS)

Core engineering principles already present and worth standardizing across modules.
Contract-First Messaging
Runtime schemas + JSON-RPC types keep sidepanel/background/relay aligned.
Bounded Session State
Maps with capacity limits prevent unbounded run/session growth.
Permission Gatekeeping
Tool category + domain allowlist checks before browser actions.
Recovery by Design
Retries for model/tool/finalization and fallback model routing paths.
Context Hygiene
Compaction/summarization keeps long sessions operational under token budgets.
Observability Surface
Run status, warnings, latency/usage benchmarks, traces, and timeline rendering.

6) How We Should Structure Next (TARGET SHAPE)

Move from file-centric orchestration to layered, bounded modules.

Target Runtime Layering

  • Surface Layer: sidepanel modules focused on rendering + local interaction only.
  • Orchestration Layer: run engine handles lifecycle/status/state transitions.
  • Domain Services: model routing, tool runtime, report-images, context compaction, account modes.
  • Adapters: browser APIs, relay bridge, Convex client, Stripe/webhooks.
  • Contracts: shared runtime messages, plan schema, JSON-RPC, tool definitions.

Concrete Refactor Slices

  • Extract run-engine from BackgroundService (status/model/tool phases).
  • Replace tool switch in browser-tools.ts with registry + common wrappers.
  • Convert handleRuntimeMessage to type → handler map modules.
  • Split account into auth, billing, setup-state, ui-render.
  • Split payments into plan-config, checkout, provisioning, recovery.
